# macOS 工作舱系统 1.0 开发需求文档

## 1. 文档信息

- **项目名称：** macOS 工作舱系统
- **英文候选名：** WorkPod / DevPod / FocusCockpit / Workspace Orchestrator / ZonePilot
- **文档版本：** V1.0
- **文档类型：** 开发需求文档
- **适用范围：** macOS 桌面端产品研发
- **目标阶段：** MVP / 1.0
- **创建日期：** 2026-03-31
- **负责人：** 李永杰

## 2. 项目背景

在 macOS 日常办公和研发场景中，用户通常需要同时使用多个应用，如 Chrome、Terminal、Word、Notes 等，并在不同任务之间频繁切换。系统原生的桌面、Spaces、Stage Manager 等能力只能提供基础窗口组织能力，无法满足以下需求：

1. 将一组应用和内容固定组织为一个"工作舱"
2. 在工作舱内进行受控切换，不轻易跳出当前上下文
3. 应用放大或全屏后仍然属于当前工作舱
4. 切换到其他工作舱时出现显性的过渡提示
5. 支持保存和恢复工作舱状态

因此，需要研发一套基于 macOS 公共能力的"工作舱系统"，实现工作场景编排、窗口控制、伪全屏、切舱提示和状态恢复。

---

## 3. 项目目标

### 3.1 总体目标
开发一款 macOS 桌面端工具，将多个桌面应用组织为可管理的"工作舱"，提供更强的任务上下文保持能力与沉浸式切换体验。

### 3.2 1.0 阶段目标
1. 支持创建多个工作舱
2. 支持为工作舱配置一组应用和资源
3. 支持一键进入工作舱
4. 支持工作舱内的窗口布局恢复
5. 支持工作舱内成员切换
6. 支持工作舱内伪全屏模式
7. 支持跨工作舱切换时的显形提示层
8. 支持基本状态保存与恢复

---

## 4. 非目标

以下内容不在 1.0 范围内：

1. 不实现系统级硬隔离
2. 不实现通用第三方原生窗口嵌入
3. 不接管 macOS 原生 Mission Control / Spaces
4. 不保证所有第三方应用原生全屏都能纳入工作舱规则
5. 不以 App Store 上架为首要目标
6. 不实现多人协作与云同步
7. 不实现跨设备状态同步

---

## 5. 用户场景

### 5.1 开发场景
用户进入"开发舱"后，系统自动打开：
- Chrome 指定开发文档页面
- Terminal / iTerm2 指定工程目录
- Word 的需求文档
- Notes 的开发笔记

用户可以在这些成员之间切换，且切换行为始终在当前工作舱上下文内完成。

### 5.2 写作场景
用户进入"论文写作舱"后，系统自动恢复：
- Word 主文档
- Chrome 参考网页
- Notes 摘录笔记

切换到其他工作舱前，系统出现显形提示层，提示将切换环境。

### 5.3 研究场景
用户进入"研究舱"后，系统恢复浏览器、PDF 工具、终端、笔记应用的组合布局，并支持将当前主窗口切换为伪全屏沉浸模式。

---

## 6. 核心概念

### 6.1 工作舱
工作舱是一个具名的任务上下文容器，由以下内容组成：
- 应用成员集合
- 启动参数
- 窗口布局
- 焦点策略
- 模式状态
- 最近快照

### 6.2 成员应用
工作舱中的每个成员应用定义一个实例配置，例如：
- Chrome + 某 profile + 多个 URL
- Terminal + 某目录
- Word + 某文档
- Notes + 某条笔记

### 6.3 模式
每个工作舱至少支持以下模式：
- `normal`：普通布局
- `immersive`：沉浸布局
- `pseudoFullscreen`：伪全屏布局

### 6.4 舱内切换
同一工作舱内成员之间切换，应保持在当前工作舱内，不应出现明显跨环境感。

### 6.5 跨舱切换
从一个工作舱切换到另一个工作舱时，必须出现显形提示层，再进入目标工作舱。

## 7. 产品功能需求

### 7.1 工作舱管理

**功能说明：** 支持用户创建、编辑、删除、复制工作舱。

**功能点：**
1. 创建工作舱
2. 设置工作舱名称、图标、快捷键
3. 添加成员应用
4. 配置成员应用的资源入口
5. 设置默认布局和默认模式
6. 删除工作舱
7. 复制已有工作舱作为模板

**验收标准：**
- 用户可通过 UI 完成工作舱的增删改查
- 配置结果可持久化保存

---

### 7.2 成员应用配置

**支持的首批应用类型：**
1. Chrome
2. Terminal
3. iTerm2
4. Microsoft Word
5. Apple Notes

**可配置项：**
- bundleId
- 启动方式
- URL / 文件路径 / 目录路径
- 布局位置
- 宽高
- 所属显示器
- 是否主成员

**验收标准：**
- 支持为每个成员配置启动参数
- 可在进入工作舱时自动打开对应资源

---

### 7.3 一键进入工作舱

**功能说明：** 用户可通过点击 UI 或快捷键一键进入工作舱。

**系统行为：**
1. 读取工作舱配置
2. 启动缺失应用
3. 连接已存在应用实例
4. 打开对应 URL / 文件 / 路径
5. 等待窗口可见
6. 执行布局恢复
7. 聚焦主成员

**验收标准：**
- 可在 1 次操作内进入目标工作舱
- 进入后主要成员应用处于可见状态
- 主成员获得焦点

---

### 7.4 工作舱内切换

**功能说明：** 在当前工作舱内切换成员应用时，不应出现明显"离开工作舱"的体验。

**系统行为：**
1. 切换焦点到目标成员
2. 必要时调整窗口层级与可见性
3. 保持当前工作舱状态不变

**验收标准：**
- 舱内切换速度快
- 不触发跨舱显形提示
- 切换后当前工作舱仍为 active 状态

---

### 7.5 伪全屏模式

**功能说明：** 成员应用可进入"伪全屏"，即由系统统一控制的沉浸模式，而不依赖第三方应用原生全屏。

**行为定义：**
1. 目标窗口放大到当前显示器工作区域
2. 其他成员窗口隐藏、缩小或移出主视野
3. 保留极简导航层或顶部状态条
4. 当前模式标记为 pseudoFullscreen

**验收标准：**
- 用户可进入和退出伪全屏
- 退出后恢复到进入前布局
- 伪全屏仍归属于当前工作舱

---

### 7.6 跨工作舱切换显形提示

**功能说明：** 从工作舱 A 切换到工作舱 B 时，必须先显示一个显形提示层，再切换。

**提示层内容：**
- 正在切换到的工作舱名称
- 目标工作舱主要成员摘要
- 可取消提示
- 可视化过渡动画

**行为流程：**
1. 冻结当前工作舱状态
2. 显示提示层
3. 恢复目标工作舱
4. 完成后关闭提示层

**验收标准：**
- 跨舱切换一定经过提示层
- 用户可在提示层中取消本次切换
- 目标舱恢复完成后提示层自动关闭

---

### 7.7 状态保存与恢复

**功能说明：** 系统应保存最近一次工作舱状态，并支持恢复。

**保存内容：**
- 当前活动工作舱
- 当前模式
- 当前焦点成员
- 每个成员窗口位置和尺寸
- 最近更新时间

**恢复内容：**
- 恢复工作舱成员应用
- 恢复窗口布局
- 恢复焦点成员
- 恢复模式

**验收标准：**
- 应用重启后可恢复上次保存的工作舱状态
- 多个工作舱可分别保存最近快照

---

### 7.8 出舱检测

**功能说明：** 系统需检测用户是否正在离开当前工作舱上下文。

**检测条件：**
1. 当前前台应用不属于当前工作舱
2. 当前焦点窗口不属于当前工作舱成员窗口
3. 检测到第三方应用进入原生全屏空间
4. 用户使用系统方式切到非当前舱目标

**处理策略：**
- 软提示：提示正在离开当前工作舱
- 强确认：弹出确认层
- 可选自动回收：拉回当前舱主成员

**验收标准：**
- 可识别明显的出舱行为
- 默认采用软提示 + 强确认策略

---

### 7.9 工作舱总览页

**功能说明：** 展示所有工作舱卡片，并支持快速切换。

**展示内容：**
- 工作舱名称
- 图标
- 最近活动时间
- 当前状态
- 成员摘要
- 缩略图预览

**验收标准：**
- 用户可从总览页切换到任意工作舱
- 当前活动工作舱有明显标识

---

## 8. 权限需求

### 必需权限
1. **Accessibility 权限**
   - 用于观察和控制其他应用窗口
2. **Screen Recording 权限**
   - 用于缩略图和预览采集

### 建议原则
- 首次启动时进行权限引导
- 未授权时提供明确提示和操作入口
- 权限不足时降级运行并提示缺失能力

---

## 9. 技术实现边界

### 可实现范围
1. 启动和激活应用
2. 控制窗口位置和尺寸
3. 跟踪前台应用与焦点窗口
4. 显示自有提示层
5. 生成窗口预览缩略图
6. 维护工作舱状态机

### 不保证范围
1. 不保证阻止用户通过所有系统手段离开当前工作舱
2. 不保证所有第三方应用的原生全屏可被完全接管
3. 不保证所有第三方应用窗口都能稳定识别
4. 不支持把所有第三方原生窗口通用嵌入到自有窗口中

---

## 10. 状态机需求

### 全局状态
- Idle
- EnteringCockpit
- ActiveCockpit
- SwitchingCockpit
- Recovering
- Error

### 工作舱状态
- Dormant
- Launching
- Restoring
- Active(normal)
- Active(immersive)
- Active(pseudoFullscreen)
- Suspended

### 关键规则
1. 全屏属于工作舱模式，不属于离开工作舱
2. 跨舱切换必须经过 overlay
3. 第三方原生全屏默认不作为主线路径
4. 只有指定热键或总览入口允许正式切舱

## 12. 非功能性需求

### 性能
1. 切舱时长应尽量控制在 2 秒内
2. 舱内切换应尽量接近即时
3. 缩略图采集不得明显影响系统流畅度

### 稳定性
1. 应用未授权时不能崩溃
2. 成员应用启动失败时应可降级处理
3. 多显示器变化时应可回退到安全布局

### 安全与隐私
1. 默认不上传任何窗口内容
2. 缩略图与预览仅本地使用
3. 日志不记录敏感正文内容

## 14. 验收标准总表

1. 用户可创建至少 3 个不同工作舱
2. 每个工作舱可配置至少 4 个成员应用
3. 一键进入工作舱可自动打开对应资源
4. 可恢复成员窗口布局
5. 同舱切换不触发显形提示
6. 跨舱切换必须触发显形提示
7. 伪全屏可进入、退出并恢复布局
8. 应用重启后可恢复最近工作舱状态
9. 未授权权限时可提示并降级运行
10. 不出现明显崩溃和不可恢复状态

---

## 15. 后续研发建议

1. 先完成 MVP，验证工作舱模型是否成立
2. 优先打磨切舱提示、伪全屏和恢复体验
3. 不建议 1.0 就强攻系统级硬隔离
4. 应以"工作舱编排器"而非"重造 macOS 桌面"定义产品边界
